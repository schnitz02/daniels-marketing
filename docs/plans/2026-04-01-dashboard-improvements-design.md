## Overview

Four improvements to the Daniel's Donuts marketing dashboard:

1. GA4 Analytics page (SEO + SEM)
2. Social Stats AI analysis (on-demand Claude commentary)
3. Ideas with evidence/reasoning
4. Research results tab

---

## Feature 1 — GA4 / SEO & SEM Analytics Page

### Goal
Surface Google Analytics 4 data so the head of marketing can track organic (SEO) and paid (SEM) search performance without leaving the dashboard.

### Architecture
- New backend client: `src/core/ga4_client.py` — wraps `google-analytics-data` Python SDK
- New API route: `src/api/routes/ga4.py` — mounted at `/api/ga4`
- Updated frontend: `dashboard/src/pages/Analytics.jsx` — replaces existing stub page
- Environment variables: `GOOGLE_ANALYTICS_PROPERTY_ID`, `GOOGLE_SERVICE_ACCOUNT_JSON` (path to service account JSON file)

### Graceful degradation
When GA4 credentials are absent from `.env`, the Analytics page renders a setup banner:
*"Connect GA4 to enable this page — add GOOGLE_ANALYTICS_PROPERTY_ID and GOOGLE_SERVICE_ACCOUNT_JSON to .env"*
No crash, no broken requests.

### SEO Tab
- Sessions from organic search (last 30 days)
- New users via organic
- Bounce / engagement rate
- Top 10 organic landing pages (page path + sessions + engagement rate)
- Line chart: daily organic sessions over 30 days

### SEM Tab
- Sessions from paid search (last 30 days)
- Conversions (goal completions linked in GA4)
- Top campaigns (if Google Ads linked)
- Cost per click trend (if Ads linked)
- Falls back to "Google Ads not linked" message if Ads data unavailable

### API Endpoints
```
GET /api/ga4/seo     → organic sessions, top pages, trend
GET /api/ga4/sem     → paid sessions, conversions, campaigns
GET /api/ga4/status  → returns { connected: bool } for frontend banner logic
```

### Data flow
Frontend → `/api/ga4/seo` → `GA4Client.get_seo_metrics()` → Google Analytics Data API → formatted response

### GA4Client behaviour
- Initialises using service account JSON at path from env var
- All methods return `None` if credentials missing (frontend handles gracefully)
- Date range: last 30 days by default
- Caches responses for 1 hour (in-memory) to avoid hammering the API

## Feature 2 — Social Stats AI Analysis

### Goal
The head of marketing can click "Generate Analysis" on any platform tab to get Claude's interpretation of the current stats — what they mean, benchmarks context, and 3 actionable recommendations.

### Architecture
- New DB model: `SocialAnalysis` — stores generated analysis per platform
- New API endpoints on `/api/social-stats`:
  - `POST /api/social-stats/analysis/{platform}` — triggers Claude, stores result
  - `GET  /api/social-stats/analysis/{platform}` — returns latest stored analysis
- Updated frontend: `SocialStats.jsx` — "Generate Analysis" button + analysis card per tab

### DB Model: SocialAnalysis
```python
class SocialAnalysis(Base):
    __tablename__ = "social_analysis"
    id            = Column(Integer, primary_key=True)
    platform      = Column(String(20))
    analysis      = Column(Text)           # Claude's full response
    generated_at  = Column(DateTime)
```

### Claude prompt structure
System: Marketing analyst for an Australian donut brand.
Input: Latest snapshot (followers, following, posts_count, engagement_rate), last 30 days of follower trend, recent post performance.
Output: JSON with keys:
- `summary` — 2-3 sentence plain-English summary of current performance
- `benchmarks` — how these numbers compare to typical food/hospitality brand averages
- `recommendations` — list of 3 specific, actionable improvements

### Frontend behaviour
- On load: fetch existing analysis (if any) and display with "Generated X minutes ago" timestamp
- "Generate Analysis" button: POST → streams response → renders result
- Button shows "Generating…" spinner while waiting
- Analysis card shows summary, benchmarks, and numbered recommendations

## Feature 3 — Ideas with Evidence/Reasoning

### Goal
Each idea generated by the Strategy agent includes the reasoning and evidence behind it, so the marketing team can make informed approval decisions.

### DB change
Add `evidence` column to `ideas` table:
```python
evidence = Column(Text, nullable=True)
```

### Strategy agent change
Update Claude prompt to return `evidence` alongside `title` and `body`:
```json
{
  "ideas": [
    {
      "title": "...",
      "body": "...",
      "evidence": "Based on [competitor insight] and [Daniel's Donuts trend], this campaign is recommended because..."
    }
  ]
}
```
Store `evidence` when creating `Idea` records.

### Frontend change
On the Approvals page, each `IdeaCard` gets a collapsible **"Why this idea? ▾"** section that expands to show the evidence text. Collapsed by default to keep cards clean.

## Feature 4 — Research Results Tab

### Goal
Display the research agent's gathered intelligence in its own dashboard page so the team can review competitor activity directly.

### New page: `Research.jsx`
- Sidebar nav entry: "Research"
- Route: `/research`

### Page layout
- **Header**: "Competitor Research" + "Run Research Agent" button (triggers research agent)
- **Filter bar**: dropdown to filter by competitor (All / Krispy Kreme / Dunkin' Donuts / etc.)
- **Feed**: research items sorted newest-first, grouped by date
  - Each card shows: competitor badge, source badge (Instagram/Web/etc.), content summary, timestamp
  - Long content truncated with "Show more" toggle

### API endpoint
```
GET /api/research/items?competitor=&limit=50
```
Returns ResearchItem records from the DB.

New route file: `src/api/routes/research.py` mounted at `/api/research`.
